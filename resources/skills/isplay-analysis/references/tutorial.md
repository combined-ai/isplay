# Deep Analysis Tutorial

## 1. Start From A Trace, Not A Theory

Begin with the referenced run. If the user mentions "that run", inspect recent artifacts or ask for the `runId`.

```bash
isplay discover run <runId>
```

Use the catalog to answer:

- Which checkpoints are available?
- Which model calls happened?
- Which tools were proposed and executed?
- Which tool schemas, arguments, and outputs entered context?
- Which state fields, retrieval chunks, or memory items were present?
- Are there open fixture requirements or prior experiments?

## 2. Build The Baseline Narrative

Write a compact baseline narrative before creating hypotheses:

- Input and task.
- System/developer/user messages that constrained behavior.
- Model settings and provider.
- Tool availability and descriptions.
- Tool call sequence and key arguments.
- Tool outputs that likely changed downstream state.
- Final response/action.

Keep this narrative factual. Mark uncertain points as hypotheses.

## 3. Select A Checkpoint

Use the earliest checkpoint that still contains enough state to replay the decision. If unsure:

- `first`: good for broad context/prompt/tool-availability interventions.
- `latest`: good for late-stage answer or final action interventions.
- `name`: best when the app provides semantic checkpoints like `claim.input` or `plan.ready`.

Avoid testing from a checkpoint after the suspected cause already happened.

## 4. Raise Hypotheses

Turn observations into testable statements:

- Bad: "The model was confused."
- Too strong: "The duplicate receipt tool output caused escalation."
- Better: "Substituting a low-risk `risk-signals` output should reduce tool-sequence distance and change the recorded/derived comparison from escalation toward approval."

Prefer many narrow hypotheses over one broad bundle. Use a batch experiment to run them together.

## 5. Run A Batch Experiment

Use one-call hypothesis batches when an analyst has several ideas. Include:

- `baseRunIds`
- `checkpointSelector`
- `hypotheses`
- `trialPlan`
- replay `policy`
- `validityGates`

Default to recorded model replay and pause-for-fixture tool replay. This preserves safety and makes missing tool outputs explicit.

## 6. Handle Fixture Requirements

If replay pauses:

1. Read the requirement.
2. Check tool name, args hash, branch, and replay.
3. Decide fixture provenance:
   - `recorded`: from original trace or imported fixture.
   - `analyst_fixture`: human supplied or deterministic domain assumption.
   - `ai_fixture`: model-simulated, weaker evidence.
   - `simulator`: generated by an explicit simulator.
4. Submit the fixture.
5. Resume or rerun the experiment.

Use branch-scoped fixtures for repeated trials over the same intervention.

## 7. Read Results

Use:

```bash
isplay experiments results <experimentId>
isplay effects list <experimentId>
```

Interpret in this order:

1. Experiment status.
2. Open requirements.
3. Trial count and comparable count.
4. Fixture dependency rate.
5. Non-comparable rate.
6. Ranked effects.
7. Diffs and first-divergence step for top effects.

## 8. Decide What You Know

Ask:

- Did the intervention change a relevant downstream metric?
- Did repeated trials agree?
- Was the result fixture-sensitive?
- Did divergence happen before or after the suspected cause?
- Are compared traces semantically aligned?
- Could model nondeterminism explain the change?

End with a bounded conclusion and the next experiment that would most reduce uncertainty.
